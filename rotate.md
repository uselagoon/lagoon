# JWT Token Generation and Rotation in Lagoon

Lagoon employs two primary types of JWT tokens for authentication and authorization: **Keycloak-backed User Tokens** and **Internal "Legacy" Tokens**.

## 1. Token Types

### Keycloak User Tokens
- **Purpose**: Authenticating end-users and service accounts for GraphQL API access.
- **Mechanism**: Standard OpenID Connect (OIDC) tokens issued by Keycloak.
- **Validation**: The `api` service validates these tokens using the `keycloak-connect` library, which checks signatures against Keycloak's public keys and verifies session validity.

### Internal "Legacy" Tokens
- **Purpose**: Service-to-service communication (e.g., `api-data-watcher-pusher` talking to the API or Auth Server).
- **Mechanism**: Symmetric HMAC (HS256) signed tokens.
- **Secret**: Uses a shared secret defined by the `JWTSECRET` environment variable.
- **Audience**: Verified against the `JWTAUDIENCE` environment variable.

## 2. Token Generation

### User Tokens
Generated by Keycloak via standard OIDC flows (e.g., Authorization Code flow for the UI, or Client Credentials for CLI/service accounts).

### Internal Tokens
Internal tokens can be generated using the `JWTSECRET`. Lagoon includes a utility script `genjwt.py` in the root directory for this purpose:

```python
# Simplified snippet from genjwt.py
import jwt
import os

payload = {'role': 'admin', 'iss': 'my-service', 'aud': os.environ['JWTAUDIENCE'], 'sub': 'my-service'}
token = jwt.encode(payload, os.environ['JWTSECRET'], algorithm='HS256')
```

## 3. Token Validation

Validation logic for internal tokens is implemented in both the `api` and `auth-server` services:

- **`services/api/src/util/auth.ts`**: Uses `jsonwebtoken.verify(token, JWTSECRET)` to validate legacy tokens. It also checks for:
    - `aud` (Audience) match.
    - `role === 'admin'` (Internal tokens must have admin rights).
    - Expiration (controlled by `LEGACY_EXPIRY_MAX` and `LEGACY_EXPIRY_REJECT`).
- **`services/auth-server/src/util/auth.ts`**: Similar validation logic using `JWTSECRET` and `JWTAUDIENCE`.

## 4. Token Rotation

### Rotating Internal Tokens (`JWTSECRET`)
Rotation of the internal shared secret is a manual configuration process:
1. **Update Secret**: Change the value of `JWTSECRET` in the environment configuration (e.g., Kubernetes Secrets, Helm values, or `.env` files).
2. **Synchronize**: Ensure the new secret is updated across ALL services that use it:
    - `api`
    - `auth-server`
    - Any background services or tasks that generate these tokens (e.g., `api-data-watcher-pusher`).
3. **Restart Services**: Restart the pods/containers for these services to pick up the new secret. 
   - *Note: Tokens signed with the old secret will immediately become invalid once the service is updated.*

### Rotating Keycloak Signing Keys
Rotation of the asymmetric keys used for user tokens is managed entirely within Keycloak. Lagoon's services dynamically fetch public keys from Keycloak's OIDC discovery endpoint, so rotating keys in Keycloak typically does not require a restart of Lagoon services, provided the client libraries handle key caching/refreshing correctly.
